# SPEC: Chat Application

**Status:** Implemented **Layer:** Interface (chat interface - reference client) **Plane:** core **Summary:** The end-user surface for conversing with tier-capped assistants in real time, across devices, in one coherent app. **Technical design:** [`DESIGN-CHAT-APP.md`](DESIGN-CHAT-APP.md) **Site section(s):** Chat application (the public end-user SPA; the counterpart to the standalone Admin Console, see [`../admin/SPEC-ADMIN-CONSOLE.md`](../admin/SPEC-ADMIN-CONSOLE.md))

> **Terminology.** This is the customer-facing chat interface, where the accepted label for a capability level is **tier** (basic, standard, premium). The platform, admin, config, and security docs use **classification** for the same concept. Both name the ceiling on data and models; "tier" in this document is the interface term, not drift.

## 1. Business problem

An end user needs to talk to an assistant to get work done, and every part of that experience has to hold together in one place. They need to sign up and get in, start a conversation at a tier they are allowed, ask a question and get a trustworthy reply, keep the thread going, bring a file or another person in, and pick up the same conversation on their phone later. The alternative is to bolt a chat UI onto a raw model and wire up identity, live delivery, sharing, and file handling themselves, or to buy a single-purpose chatbot with no multi-party, tier-governed collaboration. (Current state: the underlying pieces - a messaging substrate, an identity pool, a credential vend, a file store - are foundations, not a product on their own.)

Without a single coherent surface, the user is exposed to the seams. A reply that arrives asynchronously looks like a hang. A conversation opened on a laptop is invisible on a phone. A model the user is not cleared for is offered and then fails. A shared link lands on a conversation the recipient cannot see. A dropped WebSocket looks like the app is broken rather than reconnecting. Each seam erodes the trust that is the whole point of a conversational product.

The chat application exists to make one coherent conversational surface: authenticate the user, show them only the conversations and models their tier allows, deliver assistant and human messages live and in order, and degrade honestly (reconnecting, paused, still-thinking) instead of failing silently. It is deliberately only the end-user surface. It carries no operator capability at all; reading, moderating across conversations, and administering the platform are the separate Admin Console's job.

## 2. Personas

See [`overview/PERSONAS.md`](../../../overview/PERSONAS.md) for the canonical definitions. The chat application is the End user's primary surface, and the Assistant participates in it as a system actor.

| Persona | What they need from this feature |
|---|---|
| End user | One place to sign in, start and continue conversations capped by tier, send a message and get a live reply, mention people or the assistant, attach a file, share a conversation, and do all of it on a phone. |
| End user (as channel owner) | The conversation creator is a channel moderator: they can rename, add or remove members, archive the conversation for everyone, and (on a premium channel) turn on Battle Mode. Any member can leave. |
| The assistant (system actor) | A participant in the conversation, not a user of the app: it greets on the first turn, answers turns as its own identity capped by tier, moderates its own channel, and competes as a named side in a battle. |
| Admin / operator | Not a user of this app. Operators use the separate Admin Console. The chat app shows an admin at most a link out to that console; it contains no operator code. |

## 3. Feature catalog

The complete expected user-facing feature surface of a chat app of this kind, grouped, with what each feature is, why it matters to users, and its status. Built features are verified against the real components under `frontend/packages/chat` and `frontend/packages/shared`. Features a reader expects but the app does not have are enumerated as first-class entries so the surface is complete and honest.

**Status legend:**
- **Built** - present and working in the current tree.
- **Out of scope** - not present. Each says whether it is a *deliberate boundary* (intentionally excluded, often for security or product focus) or a *roadmap candidate* (reasonable future work, simply not built).

### 3.1 Authentication and account

- **Sign in (login).** *Status: Built.* Email-and-password sign in through Cognito on a custom, non-Amplify screen (`LoginScreen`, `@ae/shared`; `useAuth().login`). It is the front door to every conversation; a user reaches nothing until they authenticate under their own identity.
- **Sign up (registration).** *Status: Built.* Self-service registration with an email, a password, and a requested tier (`RegistrationScreen`, chat; `useAuth().register`). It lets a new user ask for access at the level they need; approval at that tier is an operator step, so registration alone does not grant access.
- **Email verification.** *Status: Built.* A one-time-code screen confirming the email, with a resend option (`EmailVerificationScreen`, chat; `confirmRegistration`, `resendCode`). It proves the address is real before an account can be approved, and the resend covers a lost or expired code.
- **Forgot-password reset.** *Status: Built.* A two-step reset (request a code, then set a new password) on a dedicated screen (`ForgotPasswordScreen`, `@ae/shared`; `forgotPassword` + `confirmForgotPassword`). It lets a locked-out user recover without an operator, which is table stakes for a self-service product.
- **Forced new-password challenge.** *Status: Built.* When an account was created with a temporary password (operator onboarding), the login screen renders a set-a-permanent-password step (`NEW_PASSWORD_REQUIRED`; `completeNewPassword`). It gets an onboarded user securely off the temporary credential on first sign in.
- **MFA challenge response.** *Status: Built.* When a sign in hits a TOTP or SMS MFA challenge, the login screen renders a one-time-code input (`SOFTWARE_TOKEN_MFA` / `SMS_MFA`; `completeMfa`). It carries a user with MFA enabled through the second factor.
- **Sign out.** *Status: Built.* A sign-out control in the header that ends the session globally (`Header`; `useAuth().logout` -> `GlobalSignOut`). It lets a user leave cleanly on a shared or public device.
- **Authenticated change-password.** *Status: Out of scope (roadmap candidate).* Changing a password while signed in. It matters for routine credential hygiene without a lockout. Today a signed-in user rotates their password through the forgot-password reset; a dedicated in-session flow is not built.
- **MFA enrollment.** *Status: Out of scope (roadmap candidate).* Enabling and setting up TOTP or SMS MFA from the app. It matters for a user who wants to add a second factor. The app only *responds* to an MFA challenge; enrolling MFA is done outside the chat app.
- **User settings / profile surface.** *Status: Out of scope (roadmap candidate).* An account page to edit display name, view tier, or manage preferences. It matters for self-service account control. The display name comes from the identity token; there is no in-app profile editor.
- **Notification preferences.** *Status: Out of scope (roadmap candidate).* Per-user control over which notifications fire and how. It matters for users who want to tune interruptions. Browser notifications are on a single request-once model with no preference surface.
- **Theme / dark-mode.** *Status: Out of scope (roadmap candidate).* A user-selectable light/dark or accent theme. It matters for comfort and accessibility. The app ships a single theme with no switcher.
- **In-app language picker.** *Status: Out of scope (roadmap candidate).* A control to change the interface language. It matters for non-English users. Internationalization scaffolding exists (i18n keys throughout, `@ae/shared/i18n`) but the language is fixed to English and a picker is an explicit future addition.

### 3.2 Conversations

- **Conversation list with unread state.** *Status: Built.* A sidebar listing the user's conversations, filtered to their tier ceiling, with a live unread indicator (`ConversationList`; `isConversationUnread` in `ConversationProvider`). It is the home base for returning to any thread, and the real-time unread mark tells the user where something new arrived without opening each one.
- **New-conversation creation (tier and model pick).** *Status: Built.* A modal to start a conversation by choosing an allowed tier, which binds the conversation to a model and a data-access and tool profile (`NewConversationModal`). It is how a user starts work at the right capability level, and offering only allowed tiers prevents starting a conversation that would then fail. The model is implicit in the tier: the router selects the actual model per intent within that ceiling.
- **Open and read a conversation.** *Status: Built.* The main pane renders a conversation's full message history with multi-day date dividers, sender labels, and long assistant messages collapsed behind a show-more control (`ConversationInterface`, `CollapsibleText`). It is the core reading experience, and the dividers and collapse keep a long or multi-day thread legible.
- **Mobile master-detail with Back.** *Status: Built.* Below a breakpoint the list and the open conversation swap as a master-detail pair with a Back affordance; on desktop both show side by side (`App` `has-active` class; `clearActiveConversation`). It makes one narrow screen usable instead of cramming two panes onto a phone.
- **Rename / title.** *Status: Built.* The conversation owner (moderator) can rename a conversation; separately, the assistant auto-derives a semantic title from the first message server-side, and the new name syncs live to every client (`renameConversation` -> Amazon Chime SDK Messaging `UpdateChannel`; the `UPDATE_CHANNEL` event). It keeps the list scannable without the user having to name every thread manually.
- **Share a conversation.** *Status: Built.* A modal to share a conversation with another user by email, adding them as a member; they open it through a deep link (`ShareConversationModal`; `?conversation=<id>`). It turns a private thread into a collaboration, and the deep link works even before the conversation appears in the recipient's list. Sharing adds an existing user; inviting a brand-new person is not a chat-app feature.
- **Archive a conversation.** *Status: Built.* The owner (moderator) can archive a conversation for everyone: it becomes read-only and hidden from every member's active list (`ArchiveConversationModal`; `archiveConversation` -> conversation-management API). It gives a finished thread a clean, shared end state that persists, rather than a per-device hide.
- **Leave a conversation.** *Status: Built.* Any member can leave, removing only their own membership (`leaveConversation`). It lets someone step out of a shared conversation they no longer need without affecting the others.
- **Conversation search / filter in the list.** *Status: Out of scope (roadmap candidate).* Searching or filtering the sidebar by title or content. It matters once a user has many conversations. The list is chronological with unread marks and no search field.

### 3.3 Messaging

- **Write and send a message.** *Status: Built.* A composer that sends the user's message and shows it immediately (optimistic), with the reply arriving live (`MessageInput`; `sendMessage`). It is the fundamental act of the app; the optimistic echo makes it feel instant.
- **Attachments (file upload and preview).** *Status: Built.* The user can drag in or pick a document or image, see it preview before sending, and send it with a message; assistant attachment replies (for example a generated document) render as downloadable files (`attachmentService`, `FileUploadPreview`, `AttachmentDisplay`). Allowed types and a 10 MB cap are enforced client-side. It lets the assistant work over the user's own file, and the preview and limits prevent a failed or oversized upload.
- **Mentions (@assistant, @member, @all).** *Status: Built.* An `@`-picker in the composer to address the assistant, a specific member, or everyone in a multi-person conversation, with a filtered dropdown (`MessageInput`, `mentionParser`). A reply targeted to the user auto-pins a sticky target so their next message continues that thread without retyping the mention. It keeps a multi-party conversation addressable and low-friction.
- **Message edit.** *Status: Out of scope (deliberate boundary).* Editing a message after it is sent. It matters for fixing a typo. Messages are immutable in the chat surface; the conversation archive is an audit trail, and letting a user rewrite history would undermine it.
- **User-side redact / delete of one's own message.** *Status: Out of scope (deliberate boundary).* Redacting or deleting a message a user sent (or any message). It matters to users who want to retract something. This is intentionally excluded: redaction and deletion are operator moderation and administration actions in the separate Admin Console (`adminChime.ts`), and the chat plane imports no message-delete or redact capability. It is a security and accountability boundary, not a gap.
- **Message reactions (emoji).** *Status: Out of scope (roadmap candidate).* Reacting to a message with an emoji. It matters for lightweight acknowledgment. Not built; the only per-message feedback is the assistant-reply thumbs used for analytics, not a social reaction.
- **Read receipts / delivery ticks.** *Status: Out of scope (deliberate boundary).* Showing whether a peer has read or received a message. It matters for multi-human conversations. The app tracks unread state for the local user's own sidebar, but it does not expose per-message read or delivery status of others, which would add cross-member presence tracking the product does not aim to provide.
- **Message search (within or across conversations).** *Status: Out of scope (roadmap candidate).* Full-text search over messages. It matters for finding a past answer. Not built on the chat surface; history is read by scrolling a conversation.
- **Threaded replies.** *Status: Out of scope (roadmap candidate).* Replying in a branching thread off a specific message. It matters for organizing a busy multi-party conversation. The model is a single linear conversation with `@`-mention targeting, not threads.
- **Message forwarding.** *Status: Out of scope (roadmap candidate).* Forwarding a message into another conversation. It matters for reusing content. Not built; sharing operates at the conversation level, not the message level.
- **Message pinning.** *Status: Out of scope (roadmap candidate).* Pinning an important message to the top of a conversation. It matters for keeping a key answer handy. Not built.
- **Mid-conversation model switcher.** *Status: Out of scope (deliberate boundary).* Changing the model partway through a conversation. It matters to power users who want to escalate. The tier chosen at creation is the real control, and the router already picks the best model per intent within that ceiling; a live switcher (a mounted `ModelSelector` exists in the tree but is not wired into the flow and reads mock data) is excluded by design so a conversation cannot silently exceed its tier.

### 3.4 Participation

- **Channel members panel.** *Status: Built.* A panel listing the conversation's members with their role (moderator, member, assistant) and initials (`ChannelMembersPanel`). It shows the user who is in the conversation, human and assistant, which matters as soon as a thread has more than one person.
- **Add and remove members.** *Status: Built.* A member is added by sharing the conversation (by email); the owner (moderator) can remove a member from the members panel (`removeConversationMember` -> conversation-management API; add via `ShareConversationModal`). It lets the owner curate who is in a shared conversation, with the mutation persisted server-side, never a local-only change.
- **Battle Mode toggle (owner, premium).** *Status: Built.* The members panel exposes a Battle Mode on/off toggle for a premium channel, choosing an armed experiment; enabling pulls the second assistant in as a visible member and makes `/battle` available (`ChannelMembersPanel` -> `channelBattleService`). It is how an owner turns an ordinary conversation into a head-to-head arena and cleanly back again.
- **Presence / online status.** *Status: Out of scope (roadmap candidate).* Showing which members are online now. It matters for knowing whether a human peer is around. Not built; membership is shown, liveness is not.
- **Peer typing indicators.** *Status: Out of scope (roadmap candidate).* Showing when another human member is typing. It matters in a multi-human conversation. Only the assistant's typing indicator exists; human peers have none.

### 3.5 Assistant interaction (system-actor features, user-visible)

- **The assistant reply.** *Status: Built.* The assistant answers a turn as its own identity, capped by the channel's tier, delivered live whether returned directly or as an async placeholder that updates in place (`ConversationProvider` message handling). It is the point of the product; the placeholder-then-update pattern means a slow, multi-step answer never looks like a hang.
- **First-turn greeting by name.** *Status: Built.* On the first real turn of a new conversation the assistant greets the user by name; the new-conversation empty state also shows tier-specific example prompts rendered client-side (not posted as a message) (`utils/greeting.ts`; server-side first-turn greeting). It gives a warm, oriented start and shows what this tier of conversation is good for, without a spurious authored message or an extra bot reply.
- **Typing / bot-typing indicator.** *Status: Built.* While the assistant is composing, a typing indicator shows until the reply lands or a timeout elapses (`isBotTyping` in `ConversationProvider`). It reassures the user that their turn was received and is being worked on.
- **Task status indicator.** *Status: Built.* For a longer-running task, an inline status chip shows the task label and an in-progress pulse (`TaskStatusIndicator`, driven by the message `activeTask`). It makes a multi-step task legible as work-in-progress rather than a stall.
- **Battle in-chat (scorecard, tally, pick a winner).** *Status: Built.* In a battle-enabled channel, `/battle <prompt>` runs two named assistants head-to-head; the chat surface renders a side-by-side scorecard (response time, an estimated cost labeled as an estimate, and a pick-the-winner control) and a running tally bar, and routes a targeted reply when an assistant is waiting on the user (`BattleScorecard`, `BattleTallyBar`, `battleTally`, `battleOutcomeService`, `channelBattleService`). It lets a user compare two assistants on their own prompt and record a preference directly. The authoring, arming, and cross-conversation results of a battle live in the Admin Console and the battle spec, not here (see [`../../capabilities/SPEC-BATTLE.md`](../../capabilities/SPEC-BATTLE.md)).

### 3.6 System and UX

- **Connection status.** *Status: Built.* A reconnecting indicator that appears only while the messaging WebSocket is disconnected (`ConnectionStatus`, driven by `MessagingProvider.isConnected`). It tells the user the app is recovering rather than broken, and disappears the moment the live connection is back.
- **Deployment status banner.** *Status: Built.* A paused banner shown when the deployment's cost-sleep mode has paused the data plane (`DeploymentStatusBanner`, gated on `VITE_SLEEP_MODE_ENABLED`). It explains a paused deployment honestly instead of letting the user hit raw timeouts; it is inert when sleep mode is not deployed.
- **Browser notifications.** *Status: Built.* When a message arrives in a conversation that is not the focused one, the app raises a browser notification (permission requested once) that routes back to the conversation on click (`notificationService`, wired through the `ConversationProvider` global listener). It brings the user back to a reply they would otherwise miss while working elsewhere.
- **Error states.** *Status: Built.* Send failures, upload failures, and invalid mentions surface inline and are dismissible, and a top-level error boundary catches render failures (`sendError` / `clearSendError`, `uploadError`, `mentionError`; `ErrorBoundary` from `@ae/shared`). Honest, local error messages let the user understand and retry instead of losing their input to a blank screen.
- **Admin link-out (admins only).** *Status: Built.* When configured, an admin sees a plain link in the header to the separate Admin Console (`Header`, `ADMIN_APP_URL`, shown only when `user.isAdmin`). It is a convenience for an operator who is also a chat user; it is only a URL, so the chat bundle still carries no operator code.
- **Voice and video calling.** *Status: Out of scope (deliberate boundary).* Real-time voice or video with the assistant or other members. It matters for some conversational products. The chat app is a text-and-attachment surface; voice/video is not a product goal.

## 4. Acceptance criteria

A new user can register at a requested tier, verify their email, and sign in with custom (non-Amplify) screens, including the forced-new-password and MFA-challenge paths, and recover via forgot-password. Once in, they can: start a conversation only at a tier they are allowed; send a message and watch the reply arrive live (direct or async) with a typing indicator; reopen a conversation with full history, day dividers, and real-time unread marks; mention a member, the assistant, or everyone; attach and send a file within the type and size limits and download an assistant attachment; share a conversation to another user who opens it by link; see the members and their roles; as the owner add, remove, rename, or archive, and as any member leave, each change persisted server-side; and run a `/battle` and pick a winner. Throughout, the app degrades honestly: a dropped connection shows a reconnecting indicator and recovers, a paused deployment shows a paused banner, a long task shows a status chip, and errors surface inline. On a phone the list and conversation work as a master-detail with Back. The chat bundle contains zero admin code and no admin endpoint URLs (the build asserts it). The out-of-scope features in the catalog are honestly absent: message edit, user-side redaction or deletion, reactions, read receipts, message search, threaded replies, forwarding, pinning, a mid-conversation model switcher, an authenticated change-password flow, MFA enrollment, a settings/profile surface, notification preferences, a theme switcher, an in-app language picker, conversation search, presence, peer typing, and voice/video.

## 5. Non-goals

- **Not the Admin Console.** The chat app never reads other people's conversations, moderates or administers content, manages users, or shows platform analytics. Those are the separate operator app ([`../admin/SPEC-ADMIN-CONSOLE.md`](../admin/SPEC-ADMIN-CONSOLE.md)). The invariant is enforced at build time (`assert-no-admin-in-chat.mjs`): the chat entry's import graph must never reach admin code.
- **Not message moderation.** End-user redaction or deletion of messages is deliberately not a chat feature; it is an operator moderation and administration capability. The chat plane carries no message-delete or redact code.
- **Not the battle authoring surface.** The chat app runs and scores a battle; it does not author experiments, arm them, or aggregate cross-conversation results. See [`../../capabilities/SPEC-BATTLE.md`](../../capabilities/SPEC-BATTLE.md).
- **Not a tier or admin management surface.** A user's tier and any admin status are set elsewhere (Admin Console user management, the identity model); the chat app only reads and respects them.
- **Not a new data store.** The app holds only session UI state; conversation history, membership, and files live in the platform stores it reads through.
- **Not streaming.** Assistant replies use the placeholder-then-update pattern, not token streaming.
- **Not a real-time media surface.** Voice, video, and presence are not product goals; the app is text and attachments.
- **Not Amplify.** Auth screens are built directly on the Cognito SDK; the app never adds Amplify.

## 6. Open product questions

- **Account self-service depth.** Whether an authenticated change-password flow, MFA enrollment, and a profile or preferences surface belong in the chat app or stay operator or identity-provider concerns is unresolved.
- **Language picker.** Internationalization scaffolding exists but the language is fixed to English; whether to ship a user-facing picker is open.
- **Messaging feature depth.** Which of the common messaging features currently out of scope (reactions, message search, conversation search, threaded replies, forwarding, pinning) are worth adding, and in what order, is an open product call; message edit and user-side deletion stay excluded as accountability boundaries.
- **Battle discoverability.** `/battle` is a typed command available only in a battle-enabled channel; whether the chat surface should advertise its availability (beyond the not-enabled hint) is open.
- **Share to a non-user.** Sharing today adds an existing user by email; whether an invite-a-new-person path belongs in the chat app or stays an operator concern is unresolved.
- **Attachment type policy.** The allowed-type list and size cap are client-side defaults; which of these a deployer should be able to tune, and where, is open.
