# DESIGN: Admin Console

**Status:** Partial (8 sections built; Aurora-only quality views ship their scoring passes incrementally) **Layer:** Interface (admin interface - reference client) **Plane:** admin **Product spec:** [`SPEC-ADMIN-CONSOLE.md`](SPEC-ADMIN-CONSOLE.md) **Summary:** A standalone React app (`@ae/admin`) whose `AdminDashboard` fans out per-tab analytics queries to admin-gated APIs, reads conversation history from the event archive, manages assistant profiles as versioned artifacts, and performs live administration actions client-side under the operator's own `${sub}-admin` identity.

## 1. Architecture

The console is a standalone frontend app in the npm-workspaces monorepo, deployed to its own origin. Its container is `AdminDashboard.tsx` (`frontend/packages/admin/src/components/admin/`), which renders a two-level navigation (a section rail plus per-section sub-tabs) over 8 sections (Overview, Conversations, Effectiveness, Models, Assistants, Experiments, Users, Security) and roughly 18 sub-views. `activeTab` (a `TabId`) is the single source of truth for content and data loading; the `SECTIONS` array only groups sub-views for the rail. The app-split mechanics (packages, invariant, CORS, deploy) are in [`DESIGN-SEPARATE-ADMIN-APP.md`](DESIGN-SEPARATE-ADMIN-APP.md); this doc covers the console's internals.

Components:

- **Container:** `AdminDashboard.tsx` - navigation, the shared date preset, `QUERIES_BY_TAB` fan-out, mode-based tab filtering, deep-link and browser-history handling.
- **Tab components** (one per sub-view, same directory): `OverviewTab`, `AlertsTab`, `LatencyTab`, `ConversationsTab` (with `MembershipTimeline`, `MessageInspectDrawer`), `EffectivenessTab`, `FlaggedResponsesTab`, `GroundTruthTab`, `ModelsTab`, `ModelStrategyTab`, `ProfilesTab`, `ExperimentsTab`, `UsersTab`, `UserManagementTab`, `MembershipAuditTab`. `EvaluationsTab`, `FlowsTab`, `TasksTab`, and `StepsTab` are retired from the nav (their detail folded into the Effectiveness drill, section 4) but their components still back that drill.
- **Chart and table primitives** (in `@ae/shared`, dependency-free SVG/CSS): `Sparkline`, `FunnelChart`, `DistributionBar`, `MetricCard`, `LineChart` (with target reference lines), `DataTable`.
- **Alerts assembler** (`alerts.ts`): a pure `computeAlerts(sources)` that scans the analytics results the dashboard already holds and returns the `bad`/`warn` alerts the Alerts tab renders; no backend query.
- **Admin services** (`frontend/packages/admin/src/services/`): `analyticsService.ts` (`queryAnalytics`), `adminConversationService.ts`, `membershipAuditService.ts`, `adminChime.ts` (the client-side administration actions), `adminAttachmentService.ts` (attachment-review S3 vend + presign), `profileService.ts` (the `manage-profiles` client), `awsConsole.ts` (infrastructure deep links), `toolRegistry.ts`, `sigv4Fetch.ts`.
- **Shared services** (`@ae/shared`): `credentialExchange.ts`, `experimentService.ts`, `feedbackService.ts`, `eventTrackingService.ts`.
- **Backend APIs** (CDK stacks): the analytics query API (`analytics-stack.ts` / `analytics-stack-aurora.ts`), admin-conversations, user-management (both in `cognito-auth-stack.ts`), experiments (`experiments-stack.ts`), membership-audit, and the user-feedback summary. The credential-exchange API vends the administration identity.

Platform fit: the console is a layer-4 **admin interface** over the layer-2 **admin foundations** (the admin APIs and the credential-exchange). It reads from the layer-1 stores (the S3 conversation archive, the Aurora/Athena analytics projection, Cognito, and the live Amazon Chime SDK Messaging substrate) but owns none of them.

## 2. Data model

The console is read-mostly and holds no store of its own. It reads:

- **Conversation event archive** (S3, queried via Athena Glue tables): the mode-independent system of record for conversation history - messages, membership create/delete, moderator grant/revoke, redaction events. Read by `admin-conversations.ts`. Present in both analytics modes.
- **Analytics projection** (Aurora Postgres in Aurora mode; Athena Glue tables in Athena mode): derived quality signals - `exchanges`, `evaluation_results`, `intent_flows`, `task_metrics` / `task_details` / `task_timeline`, `execution_steps` (from message metadata), `drift_events` (pgvector), `model_effectiveness`, `experiment_results`. Aurora-only where a query needs relational joins or pgvector.
- **Client-events Glue table** (`client_events`, Athena-served in both modes): the `/events` ingestion path populates it; backs `active_users_daily`, `active_messaging_users_daily`, `error_rate_daily`, `page_load_metrics`, `connection_health_daily`, `messages_per_user`, and the sign-up / sign-in funnels.
- **Cognito user pool:** users and their derived status (pending / approved / disabled) plus classification groups, via `user-management.ts`.
- **Live Amazon Chime SDK Messaging:** current membership (`ListChannelMemberships`), read client-side under the `${sub}-admin` identity, not from the archive.
- **Membership-audit table** (DynamoDB): Layer-6 findings and the `config/enforce` toggle item.
- **Feedback table** (DynamoDB): thumbs and approval, aggregated over a trailing 30-day window.
- **Conversation attachments** (S3): a message's stored attachment objects under `generated-docs/<channelId>/*` (assistant deliverables) or `attachments/<channelId>/*` (user uploads), read only through short-lived, channel-scoped, audited credentials vended per open. The console holds no standing S3 access.
- **Assistant-profile registry** (SSM version store, read/written through the `manage-profiles` API): each profile's versions, active pointer, draft, and resolved config body. Model keys are catalog keys, never ARNs, so the manifest is instance-agnostic.
- **Metric targets:** `metricTargets.ts` (in `frontend/packages/admin/`) is the display-only registry of good / warn / bad bands, also the basis the Alerts assembler reads.

## 3. APIs and interfaces

**Analytics query API** (`queryAnalytics(queryType, dateRange, extra?)`): a single POST endpoint that dispatches on `queryType`. `QUERIES_BY_TAB` in `AdminDashboard.tsx` maps each tab to the query types it requests; `loadData()` fans them out with `Promise.all`. Aurora-only query types are still requested in Athena mode - the backend returns `200 + { unsupported: true, reason }`, and the dashboard renders an honest banner rather than a silently empty table. The date preset (`7d` / `30d` / `90d`) yields `{start, end}` via `getDateRange`; the backend validates ISO dates, requires `start <= end`, and caps the window at 365 days.

**Admin-conversations API** (`admin-conversations.ts`, GET, admin-gated, origin `adminAppUrl`): `listConversations` (latest name per `ChannelArn`, last activity, classification from the archive partition) and `listMessages` (latest row per `MessageId`, body decoded and unwrapped from the Lex envelope). Holds no Amazon Chime SDK bearer.

**User-management API** (`user-management.ts`, GET/POST, every route admin-gated by `callerIsAdmin` and origin-checked): `GET /users`, `POST /approve` (sets approved and classification, syncs the single classification group, enables, provisions the member Amazon Chime SDK Identity app-instance-user), `POST /reject`, `POST /enable`, `POST /tier` (validates and syncs the classification group), `POST /delete` (best-effort channel-membership cleanup as the service app-instance-admin, then delete the app-instance-user and the Cognito user; idempotent).

**Experiments API** (`experiments-stack.ts`, shared origin): `listExperiments()` (registry), `experiment_results` (Aurora), create / pause / resume / complete, and a per-experiment recommendation (an LLM call over collapsed per-variant rows; descriptive only).

**Profiles API** (`manage-profiles`, `POST admin/profiles` on the experiments API; `profileService.ts`): list profiles with their versions/active pointer/draft, read a version's resolved config body, create a version, edit/validate/activate a draft, roll back, and export/import a profile. Every mutation is gated server-side on the `manage-profiles` capability (A14) and audited; under `adminIamEnforcement` the routes are `AWS_IAM` and the client SigV4-signs with the operator's sign-on credentials, else it falls back to the Cognito Bearer path. `awsConsole.ts` builds the profile's AWS infrastructure deep links client-side from server-resolved identifiers.

**Attachment-review vend** (`adminAttachmentService.ts`, credential-exchange `identity:'admin'`): to open a conversation attachment the console vends short-lived, channel-scoped, audited S3 credentials whose session policy permits `s3:GetObject` on only the named channel's attachment keys, then presigns the GET client-side and opens it as a top-level navigation. Two capabilities split by key prefix so the sensitivity boundary is IAM-enforceable: `attachment-read` (`generated-docs/…`, assistant deliverables, archive grade) and `attachment-read-uploads` (`attachments/…`, user uploads / PII, moderation grade + sensitive audit). A key under neither prefix is rejected. No standing server-side S3 bearer.

**Membership-audit API** (`membership-audit-admin.ts`, admin-gated by `requireAdmin`): `GET`/`POST /membership-audit/enforce` (report-only vs auto-revoke, reads/writes the DynamoDB `config/enforce` item), `POST /membership-audit/revoke` (`DeleteChannelMembership` as the service app-instance-admin, marks the finding revoked; idempotent). The Layer-6 construct is always deployed (in whichever analytics mode), so this surface is always present.

**Credential-exchange API** (`plane:'admin'`): vends a fresh, single-channel-scoped, short-lived, audited credential returning the operator's `${sub}-admin` ARN as the `ChimeBearer`, plus the requested `capabilities`. Called client-side per administration action. Dual-plane (both apps call it), so its CORS allows both origins.

**Administration actions** (client-side, `adminChime.ts`, each vending a per-action credential and emitting a telemetry event):

| Action | Amazon Chime SDK call | Capability |
|---|---|---|
| List members | `ListChannelMemberships` (live) | `view` |
| Join (visible / invisible) | `CreateChannelMembership` (DEFAULT / HIDDEN) | `manage-membership` |
| Add member | `CreateChannelMembership` for an ARN | `manage-membership` |
| Remove member | `DeleteChannelMembership` | `manage-membership` |
| Redact | `RedactChannelMessage` (soft; body cleared, message remains) | `redact` |
| Delete | `DeleteChannelMessage` (hard, irreversible) | `delete` |

## 4. Key flows and algorithms

**Per-tab load (FR-1..FR-7).** On tab switch or date change, `loadData()` reads `QUERIES_BY_TAB[activeTab]`, fans out `queryAnalytics` per query type, and merges results into a keyed map. The banner logic inspects only the *current* tab's queries (not the union of all prior tabs) to classify the state: `allFailed` (API down), `unsupported` (mode does not serve this query, only bannered when no sibling query returned data), or `no rows yet` (empty state). A tab with any real data renders and lets each widget's own empty state stand.

**Conversation read and administer (FR-2, FR-10).** The Conversations tab reads history from the archive via `admin-conversations.ts` (that handler holds no bearer). The full conversation view combines two sources by design: the event archive for a faithful audit trail (messages, membership, moderator, redaction), and the analytics projection for derived context (running summary, cross-conversation context, evaluations, drift, per-turn steps). Live members and every mutation run client-side: the action vends a fresh `${sub}-admin` credential from the credential-exchange (`plane:'admin'` plus the requested `capabilities`), calls Amazon Chime SDK Messaging directly from the browser, and emits a telemetry event. Fail modes: an unauthorized operator (not in `admins`) never reaches the console (entry gate); a credential-exchange denial fails the single action, not the read; Amazon Chime SDK Messaging has no membership-history API, so the archive-derived timeline is the system of record for who joined or left and when.

**Effectiveness drill (FR-3, Aurora).** The dashboard reads `intent_effectiveness` (one row per intent) and ranks worst-first on two separated axes - Classification (average confidence + reroute rate) and Execution (DIRECT intents by Pass A relevance, task intents by completion rate) - with latency, cost per reply (tokens times model rate, null-honest), and tool-error rate (from per-step `tools[]` outcomes) as independently sortable columns, colored against `metricTargets.ts`. Clicking anywhere on a row (whole-row drill, essential on touch) drills L1 (metric cards) to L2 (`intent_exchanges` or `task_details?intent`) to L3 (`task_timeline`, per-task turn-by-turn state) to L4 (that turn's tool-loop steps, inline). Consolidation is complete: the detail from Evaluations (L2 per-exchange judge verdict), Flows (L3 `FlowScorePanel`), Tasks (L2 task list + L3 timeline), and Steps (L4 inline) lives entirely inside the drill, so those four standalone sub-tabs are retired from the nav with no information lost; only the two human-action views (Flagged, Ground Truth) remain their own sub-tabs. Scores are context-aware and marker-stripped with the same deterministic `stripMessageMarkers` the SPA uses, so the judge scores exactly what the human saw.

**Alerts assembly (FR-1, health).** The Alerts tab loads `intent_effectiveness`, `latency_metrics`, `model_effectiveness`, `flagged_responses`, and `error_rate_daily` and passes them to the pure `computeAlerts(sources)` (`alerts.ts`), which returns every measured value in `bad` (an error) or `warn` (approaching threshold) against its `metricTargets.ts` band, plus literal `error_response` rows - across three categories (runtime error, latency SLA breach, quality breach). Errors sort before warnings; each row states value vs target and links to its drill (a quality alert to Effectiveness, a latency alert to Latency, a runtime error-response alert to Flagged, the error-rate alert to Overview). It leans on the Aurora-only effectiveness/model data, so the tab is Aurora-only. No new backend query.

**Assistant-profile lifecycle (FR, Assistants).** `ProfilesTab` lists each profile's versions, active pointer, and draft via `profileService.listProfiles()`, loads the active version's resolved config body for the read-only values view, and drives create / edit-draft / validate / activate / roll back / export / import through the `manage-profiles` API. Rich draft editors cover the models bundle (base, classifier, complex, and per-intent routes; the classifier default is the explicit `'default'` sentinel that tracks the platform default rather than pinning a key) and the tool set. `awsConsole.ts` renders infrastructure deep links, and a link jumps to the Effectiveness tab scoped to the profile's classification.

**Attachment review (Conversations).** From the Info drawer, opening a message's attachment calls `adminAttachmentService.getAdminAttachmentDownloadUrl(channelArn, fileKey)`, which vends channel-scoped, short-lived, audited S3 credentials for exactly that key (capability chosen by prefix: `attachment-read` for deliverables, `attachment-read-uploads` for user uploads), presigns the GET client-side, and opens it. A non-attachment key or a capability the operator lacks fails the single open, not the read.

**Model comparison and experiments (FR-4, FR-5).** `listExperiments()` reads the registry; `experiment_results` (Aurora) aggregates per variant weighted by exchange count, deriving cost from average tokens and model price, folding in feedback and battle stores. The advisory objective is evaluated client-side against the experiment target; the recommendation LLM call is descriptive and never reroutes traffic. Create supports an optional `/battle` config (display names, prompt addenda, per-variant image-gen model, alt-bot slot, long-form mode).

**Turn debugging (FR-4, FR-7 debug).** `execution_steps` (Aurora) reads each message metadata's `steps` array (newest first); each step carries `tools: Array<{name, ok, errorClass}>` with a bounded `errorClass` (`timeout` / `not_found` / `unauthorized` / `bad_input` / `error`, never raw text or PII), which feeds the Effectiveness `tool_error_rate` and renders inline at drill L4. The inspect drawer renders every stored field plus the raw archived `Payload`.

**Membership audit (FR-7).** An auditor Lambda consumes the same Amazon Chime SDK Messaging-to-Kinesis stream as the archive, filtered to membership create/update events, and flags cross-classification leaks: a **member** violation is a human whose classification ranks below the channel's; an **assistant** violation is a per-classification bot whose classification ranks above the channel's (which Layer-1's IAM tag gate does not stop). Channel classification reads the immutable `classification` tag, not the mutable `modelTier` metadata. A bot ARN that matches no classification (for example a `/battle` alt-slot) is left alone. On a violation the auditor logs a structured audit event, persists the finding, and alerts admins. When enforcing, auto-revoke runs from the auditor itself as the service app-instance-admin; the UI shows only open findings, closed by revoking.

**Deep-linking and history.** `?admin=<tab>` deep-links to a sub-tab; `?conv=<channelId>` opens a conversation detail (the short channel id is expanded to a full ARN via `VITE_APP_INSTANCE_ARN`). Tab selection pushes history so browser Back steps through tabs; a tab with an open drill-down registers a drill-back handler so Back closes the detail first rather than jumping past the list.

## 5. Security and IAM

- **Entry gate.** The admin app gates on the `admins` group via the `isAdmin` claim the shared `AuthProvider` decodes; a non-admin who authenticates sees an access-denied screen, never the console.
- **Backend gate.** Every admin endpoint routes through `callerIsAdmin` / `requireAdmin` (`backend/lambda/src/lib/auth.ts`), honoring `ADMIN_GROUP_NAMES` and service mode, and is origin-checked. The gate is independent of which frontend called it.
- **Administration identity, not chat identity.** Live actions never run as the operator's chat `${sub}`; they run as the per-human `${sub}-admin` app-instance-admin identity, vended per action via credential-exchange `plane:'admin'` as a single-channel-scoped, short-lived, audited credential. A separate no-human **service** app-instance-admin covers automation (Layer-6 auto-revoke, delete-user cleanup). No server-side bearer swap for human actions. Authority model: [`SPEC-ADMIN-IDENTITY.md`](../../interaction/identity-access/admin/SPEC-ADMIN-IDENTITY.md).
- **Classification (not tier) scoping.** Classification decisions (membership audit, per-classification bot resolution, channel clearance) read the immutable `classification` tag, never the mutable metadata.
- **Injection defense.** Displayed messages and evaluated text are marker-stripped with the deterministic `stripMessageMarkers`, so the console never renders a raw control marker and the judge scores what the human saw.
- **Attachment review is vended, not proxied.** The console never holds standing S3 access; each attachment open vends a short-lived, channel-scoped, audited `s3:GetObject` session policy for exactly one key. User uploads (`attachment-read-uploads`) are moderation-grade with a sensitive-audit record, distinct from the archive-grade `attachment-read` for assistant deliverables.
- **Per-capability IAM (A14).** Under `adminIamEnforcement` each admin action (analytics per capability, admin-conversations reads, user management, experiments, `manage-profiles`, the attachment vends, and redact / delete / membership) is IAM-enforceable per capability, so a role can be denied a specific action; four example persona roles are opt-in behind `enableAdminPersonas`, and the dedicated admin app-client is where they attach. With the flag off the `admins`-group gate is the control, unchanged. Design: [`DESIGN-ADMIN-ACTION-IAM-ENFORCEMENT.md`](../../interaction/identity-access/admin/DESIGN-ADMIN-ACTION-IAM-ENFORCEMENT.md).

## 6. Testing

- **Unit (Vitest, `frontend/packages/admin/src/components/admin/` and `.../services/`):** `EffectivenessTab.test.tsx` (the intent-anchored dashboard and drill), `alerts.test.ts` (the `computeAlerts` assembler), `FunnelChart.test.tsx` (sign-up / sign-in funnel primitive), `ConversationsTab.back.test.tsx` and `ConversationsTab.redaction.test.tsx` (drill-back history and redaction rendering), `ExperimentsTab.objective.test.ts` (objective evaluation and recommendation), `adminAttachmentService.test.ts` (the attachment capability/prefix mapping and vend). Backing analytics SQL is unit-tested at the query-dispatch layer.
- **Integration:** the admin APIs are exercised through their handlers (analytics query dispatch, `user-management.ts`, `membership-audit-admin.ts`, `manage-profiles.ts`) in the backend test suite; the credential-exchange S3-vend path has its own handler tests.
- **End-to-end (Playwright, `tests/e2e/`):** `admin-dashboard.spec.ts`, `admin-nav.spec.ts`, `admin-dashboard-render.spec.ts` (console load, section/sub-tab navigation, render); `admin-profiles.spec.ts` (the profile lifecycle) and `admin-attachments.spec.ts` (attachment review); `experiments.spec.ts` and `battle.spec.ts` (experiment lifecycle and `/battle`); `drift-detection.spec.ts` and `task-state-machine.spec.ts` and `tasks.spec.ts` (the quality data behind the Effectiveness drill); `feedback.spec.ts` (the feedback summary); `credential-exchange.spec.ts` (the administration-identity vend).
- **Deferred / gaps:** Flagged and Ground Truth backing queries are not yet served in either analytics mode (the tabs render their empty states, and their review / submit forms post but have no populated read). The Flows scoring pass ships but `intent_flows` is empty until it runs against a deployment. Non-Cognito admin federation is designed for but not validated end to end.

## 7. Migration / phasing / rollout

- **Analytics mode.** Athena is the default; Aurora is opt-in (`--context analyticsMode=aurora`). Aurora is a strict superset - every Athena view is served in Aurora, and Aurora adds the quality views; Aurora-only sub-tabs are hidden in Athena mode.
- **Layer-6 always on.** The membership audit is always deployed; the enforce default is off (`membershipAuditEnforce`, report-only) and is overridden at runtime by the `config/enforce` toggle.
- **App split.** The console ships as a standalone app; deploy phasing and stack ordering are in [`DESIGN-SEPARATE-ADMIN-APP.md`](DESIGN-SEPARATE-ADMIN-APP.md).
- **Consolidation (complete).** Evaluations, Flows, Tasks, and Steps have folded into the Effectiveness drill (L2 exchanges / L3 `FlowScorePanel` / L2 tasks + L3 timeline / L4 inline steps) and are retired from the nav; their components still back the drill. Only Flagged and Ground Truth remain standalone human-action sub-tabs.

## 8. Open technical questions

- **Flagged / Ground Truth backing queries.** Which store and query shape serve the flagged-response and ground-truth reads in each analytics mode is unresolved (only the write path exists today).
- **Per-capability IAM enforcement.** The A14 follow-up needs the capability taxonomy and the session-policy shape finalized before actions can be denied per capability rather than per group.
- **Consolidation without loss.** The exact point at which a standalone sub-tab (Evaluations, Flows, Tasks, Steps) can be removed once its detail lives in the drill needs a per-view checklist so no information is lost.
- **Athena approximations.** Whether any Aurora-only quality view warrants an approximate Athena rendering versus staying honestly gated.
