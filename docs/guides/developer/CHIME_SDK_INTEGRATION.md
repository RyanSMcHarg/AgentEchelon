# Chime SDK Integration

## Overview

The frontend talks to AWS through Amazon Chime SDK Messaging: users authenticate
with Cognito, exchange their identity for temporary AWS credentials, and use those
credentials to create channels, send messages, and receive real-time updates.
Assistant responses are generated by AWS Bedrock and delivered back into the
channel as persistent messages.

## Components

### SDK dependencies

- `@aws-sdk/client-chime-sdk-messaging` - channel/message operations
- `@aws-sdk/client-chime-sdk-identity` - user identity management
- `@aws-sdk/client-cognito-identity` - credentials
- `@aws-sdk/credential-provider-cognito-identity` - auth integration

### Environment configuration

`frontend/.env` holds the AWS resource identifiers the bundle needs, sourced from
CDK stack outputs - App Instance ARN, Identity Pool ID, User Pool ID and Client
ID, and the API URLs. See `frontend/.env.example` for the full list.

### Chime SDK service layer

`frontend/src/services/chimeService.ts` wraps the SDK:

- **Initialize** - sets up the SDK clients with Cognito-vended credentials
- **Create Conversation** - creates Chime channels with model metadata
- **List Conversations** - fetches the user's channels
- **Send Message** - posts messages to channels
- **List Messages** - retrieves channel message history
- **User Management** - creates/manages App Instance Users

### Auth integration

`AuthProvider.tsx` exposes the Cognito `idToken` on its context. The token is
retrieved on login, cleared on logout, and consumed by the client/messaging
providers to initialize the Chime SDK.

### Conversation & messaging providers

The provider hierarchy is `AuthProvider вҶ’ AwsClientProvider вҶ’ MessagingProvider вҶ’
ConversationProvider`. `MessagingProvider` owns the Chime WebSocket session;
`ConversationProvider` handles conversation and message state, including
tier-based model filtering when a channel is created.

## Real-time messaging

New messages arrive over a Chime SDK Messaging WebSocket session managed by
`MessagingProvider`. On visibility-stale reconnects the provider refreshes
credentials before re-opening the socket so the handshake uses fresh credentials.

## Aurora Analytics Mode (optional)

The default analytics pipeline uses Kinesis вҶ’ Firehose вҶ’ S3 вҶ’ Athena. For advanced
features, an optional Aurora PostgreSQL mode provides:

- **Multi-turn flow evaluation** - Groups exchanges by task ID, scores holistically across 5 weighted dimensions
- **Drift detection** - Detects when conversations shift from their stated purpose using LLM + SQL comparison
- **Cross-conversation context** - Finds related prior conversations for a user via topic matching / pgvector
- **Flagged response review** - Admin review queue with approve/reject workflow
- **Ground truth calibration** - Human scoring alongside automated evaluation, with MAE and agreement metrics
- **Task completion tracking** - Lifecycle metrics per task type (completion rate, duration, abandonment)
- **Conversation summaries** - Auto-generated purpose, topics, key points with versioning

Enable at deploy time:
```bash
npx cdk deploy --all --context analyticsMode=aurora
```

The admin dashboard automatically shows additional tabs (Flows, Flagged, Ground
Truth, Tasks, Steps) when Aurora mode is detected. See
[SPEC-AURORA-VPC-MODE.md](../../specs/analytics-eval/SPEC-AURORA-VPC-MODE.md) for full architecture details.
